# FEAT-0001-ISSUE-005: Токен передаётся в URL WebSocket — потенциальная утечка в логи

## Severity
minor

## Файл
`frontend/lib/ws.ts` строка 26–27

## Проблема

Access token передаётся как query-параметр в URL WebSocket-соединения:

```typescript
const tokenParam = token !== null ? `?token=${token}` : "";
const url = `${WS_BASE_URL}/ws/${this.workspaceId}${tokenParam}`;
```

URL WebSocket-соединения:
1. Попадает в логи веб-сервера (Nginx, Caddy, и т.д.) в секцию access log в открытом виде
2. Виден в DevTools → Network в поле "Request URL"
3. Может быть перехвачен промежуточными прокси, которые логируют URL

Это нарушает принцип безопасности из CLAUDE.md: "Секреты НЕ попадают в: логи".

Контракт в DESIGN-01.md также указывает URL как `{NEXT_PUBLIC_WS_URL}/ws/{workspaceId}?token={getToken()}` — то есть это осознанное архитектурное решение. Тем не менее риск утечки через server-side логи реален.

Дополнительная проблема: токен не кодируется через `encodeURIComponent`. JWT-токены содержат символ `.` (точка), который безопасен в query-параметрах, но могут содержать `+`, `=`, `/` в base64-секции если алгоритм подписи нестандартный. Отсутствие encode может вызвать ошибки парсинга URL на сервере.

## Ожидаемое поведение

1. Токен должен передаваться через `encodeURIComponent`:
```typescript
const tokenParam = token !== null ? `?token=${encodeURIComponent(token)}` : "";
```

2. В рамках MVP это приемлемо (архитектурное решение зафиксировано). Для будущих итераций рекомендуется переход на передачу токена в первом WebSocket-сообщении после установки соединения (протокол уровня приложения), что исключает попадание токена в transport-layer логи.

## Требует пересмотра архитектуры?
Нет — для MVP достаточно добавить `encodeURIComponent`. Переход к передаче токена в первом сообщении — отдельная задача.
