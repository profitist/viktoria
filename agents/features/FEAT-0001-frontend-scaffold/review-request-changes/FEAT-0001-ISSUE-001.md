# FEAT-0001-ISSUE-001: Mutex для refresh не защищает от race condition

## Severity
critical

## Файл
`frontend/lib/api.ts` строки 84–92

## Проблема

Реализация mutex содержит race condition. Рассмотрим сценарий:

1. Запрос A получает 401, `refreshPromise === null`, запускает `doRefresh()` → устанавливает `refreshPromise = promise`.
2. `doRefresh()` завершается (успешно или с ошибкой), `.finally()` устанавливает `refreshPromise = null`.
3. Запрос B (параллельный) получает 401 чуть позже — к этому моменту `refreshPromise` уже `null`.
4. Запрос B запускает **второй** `doRefresh()`, хотя первый уже завершился.

Это происходит потому что `.finally()` обнуляет `refreshPromise` до того, как все ожидающие запросы успевают его прочитать. В реальной среде браузера микротаски обрабатываются быстро, и промежуток между `await refreshPromise` одного запроса и `refreshPromise === null` другого минимален — но не нулевой.

Дополнительная проблема: если `doRefresh()` упал с ошибкой, то `refreshPromise` становится rejected promise. Запрос B, который уже успел сохранить ссылку на него, получит ошибку — это корректно. Но запрос C, который пришёл **после** `.finally()`, снова запустит `doRefresh()` несмотря на то что сессия уже протухла — и снова вызовет `onLogout?.()`.

Таким образом при двух параллельных 401 возможны два вызова `onLogout()`, что может вызвать двойной редирект или двойную очистку sessionStorage.

## Ожидаемое поведение

Классический паттерн: вместо обнуления `refreshPromise` в `.finally()` нужно обнулять его только **после** того, как все ожидающие `await refreshPromise` завершились. Один из способов — хранить промис и не обнулять его немедленно, или использовать reference-counting. Наиболее простое решение: обнулять `refreshPromise` **только при следующем запросе**, а не в `.finally()`.

```typescript
// Вариант: не обнулять в finally, а проверять resolved-статус
if (refreshPromise === null) {
  refreshPromise = doRefresh().finally(() => {
    refreshPromise = null; // ← убрать отсюда
  });
}
await refreshPromise;
// ← сюда перенести: refreshPromise = null; НО это тоже не решает race
```

Правильное решение: сбрасывать `refreshPromise = null` после того, как retry-запрос отправлен, либо гарантировать, что `doRefresh()` вызывается максимум один раз через атомарную проверку перед установкой (что и делает текущий код, но теряет защиту из-за async gap).

Реальный рабочий паттерн — не обнулять `refreshPromise` немедленно, а обнулять его после повторного запроса:

```typescript
try {
  if (refreshPromise === null) {
    refreshPromise = doRefresh();
  }
  await refreshPromise;
} finally {
  refreshPromise = null; // обнуляем после того как все ожидающие прошли
}
```

Но и это не атомарно. Единственно надёжный вариант для JavaScript (однопоточный) — хранить промис до завершения **всего цикла** retry, то есть не обнулять до момента отправки повторного запроса.

## Требует пересмотра архитектуры?
Нет — требует только исправления логики в `apiFetch`, без изменения структуры модулей.
