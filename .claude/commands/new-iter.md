# Новая итерация

Твоя задача — спроектировать пул задач для следующей итерации проекта. Атомарных, непересекающихся, с чёткими acceptance, готовых к параллельной работе нескольких людей.

Пользовательский ввод (опциональная цель/направление итерации): $ARGUMENTS

## Алгоритм

### Шаг 1. Проверь наличие `project/PROJECT.md`

Если файла нет — это первый запуск, проведи короткий **установочный диалог** через `AskUserQuestion`:

- **Vision:** что строим, для кого, какую боль решаем
- **MVP scope:** что точно входит в первую версию, что точно НЕ входит
- **Стек:** язык / фреймворк фронта / фреймворк бэка / БД / деплой
- **Модули верхнего уровня:** список модулей с одной строкой ответственности каждый (например `auth — регистрация/логин/сессии`, `profile — данные пользователя`, `feed — лента контента`)
- **Контракты модулей:** для каждой пары взаимодействующих модулей — название API, формат payload (markdown-описанием, без формальных схем)
- **Соглашения:** ветки задач `T-XXX-slug`, коммиты начинаются с `T-XXX:`, PR закрывает issue через `Closes #N`

Сохрани результат в `project/PROJECT.md` (создай папку `project/` если её нет). Шаблон секций:

```markdown
# Проект <название>

## Vision
...

## MVP Scope
**В:** ...
**Не в:** ...

## Стек
- Frontend: ...
- Backend: ...
- БД: ...
- Деплой: ...

## Модули
- **auth** — ...
- **profile** — ...

## Контракты модулей
### auth → frontend
**POST /api/auth/login** — `{ email, password }` → `{ token, user_id }`
...

## Соглашения
- ветки: `T-XXX-slug`
- коммиты: `T-XXX: краткое описание`
- PR: `Closes #N` в описании
```

Покажи драфт пользователю и получи подтверждение перед сохранением.

### Шаг 2. Прочитай контекст

- `project/PROJECT.md` — модули и контракты
- `project/KANBAN.md` — если существует, найди последнюю итерацию и список её задач (статусы и files_touched)

Определи номер новой итерации:
- если `KANBAN.md` нет — это **I-01**
- иначе — следующий после максимального `I-XX` в файле

### Шаг 3. Сформируй пул задач

Через диалог с пользователем (можно через `AskUserQuestion`):

1. **Цель итерации** — какое наблюдаемое поведение должно работать после её завершения. Одно-два предложения.
2. **DoD итерации** — список того, что должно работать в конце (smoke-tests, опиши конкретными командами или действиями: `npm run dev → открыть /login → войти под email/password`).
3. **Backlog задач.** Для каждой задачи зафиксируй:
   - `id` — формат `T-XXX` с продолжением нумерации (если в KANBAN.md последняя задача T-007 — следующая T-008)
   - `title` — короткое
   - `module` — из списка модулей в PROJECT.md
   - `files_touched` — конкретные пути файлов, которые задача создаёт или редактирует
   - `depends_on` — список id задач, которые должны быть выполнены раньше (только id из этой или предыдущих итераций)
   - `acceptance` — буллет-лист проверяемых критериев

### Шаг 4. Правила декомпозиции

Применяй **жёстко**, отказывайся принимать задачи, нарушающие правила:

1. **Непересекаемость:** внутри одной итерации `files_touched` любых двух задач не должны пересекаться. Если пересекаются — переформулируй разбиение или перенеси одну в следующую итерацию.
2. **Один модуль на задачу:** задача трогает файлы одного модуля. Если нужно трогать два — это либо контракт (фиксируется в PROJECT.md, а не задачей), либо две задачи.
3. **Первая итерация (I-01)** всегда содержит:
   - scaffolding каждого модуля (пустые папки + минимальный entrypoint)
   - типы/интерфейсы по контрактам из PROJECT.md как код (в одном файле на модуль)
   После I-01 модули можно пилить параллельно.
4. **Атомарность acceptance:** каждая задача завершается работающим (хотя бы изолированно) куском. Acceptance проверяемы (нет «работает корректно» — есть «возвращает 200 на POST /api/foo с body=bar»).
5. **Без циклов:** проверь `depends_on` — не должно быть цикла, зависимости только на задачи прошлых итераций или на более ранние задачи этой же итерации.
6. **Минимально достаточно:** 4-8 задач на итерацию для хакатона. Больше — итерация становится не атомарной.

Если правила нарушены — покажи пользователю проблему и предложи фикс.

### Шаг 5. Покажи драфт и получи подтверждение

Покажи пользователю:
- цель и DoD итерации
- таблицу задач (id, title, module, files_touched, depends_on)
- список найденных конфликтов (если были и как ты их разрешил)

Получи `да` перед записью в файлы.

### Шаг 6. Запиши в `project/KANBAN.md`

Если файла нет — создай с заголовком `# Kanban`.

Дописывай **сверху**, активная итерация всегда первой. Формат блока — только таблица, без acceptance details (они живут в Issue):

```markdown
## Iteration I-XX (active) — Цель: <цель>

**DoD:** <одна строка — что должно работать в конце>

| ID | Title | Module | Owner | Status | Issue | Files |
|----|-------|--------|-------|--------|-------|-------|
| T-XXX | ... | auth | — | todo | — | src/auth/foo.ts, src/auth/bar.ts |
| T-XXY | ... | profile | — | todo | — | src/profile/page.tsx |
```

Предыдущие итерации остаются ниже без изменений. Если предыдущая ещё `active` — переведи её в `closed` только после явного подтверждения (она могла быть не завершена).

### Шаг 7. Создай GitHub Issues (если доступно)

Проверь доступность через `gh auth status` и `gh repo view`. Если `gh` настроен и репо привязан:

Для каждой задачи:

```bash
gh issue create \
  --title "T-XXX: <title>" \
  --body "<body из шаблона ниже>" \
  --label "iter:I-XX,module:<module>,status:todo"
```

Тело issue — полное техзадание:

```markdown
**Iteration:** I-XX | **Module:** <module> | **Depends on:** T-XXX (или —)

## Context
<Почему эта задача существует: что она разблокирует, какую часть системы закрывает>

## What to do
<Конкретные шаги реализации: что создать, что заполнить, на что ориентироваться из PROJECT.md>

## Files
- `path1` — <создать/изменить: что именно здесь должно быть>
- `path2` — <создать/изменить: что именно здесь должно быть>

## Acceptance
- [ ] <конкретная проверка: команда, curl, действие в UI — с ожидаемым результатом>
- [ ] <например: `curl -X POST /api/auth/login -d '{"email":"a@b.com","password":"x"}' → 200 {"token": "..."}` >

## Notes
<Архитектурные ограничения, подводные камни, ссылки на контракты из PROJECT.md — если есть>

---
Ветка: `T-XXX-slug` | После готовности: PR с `Closes #N`
```

При первом запуске убедись, что лейблы существуют — создай недостающие:

```bash
gh label create "iter:I-XX" --color "0E8A16" 2>/dev/null || true
gh label create "module:<module>" --color "1D76DB" 2>/dev/null || true
gh label create "status:todo" --color "FBCA04" 2>/dev/null || true
gh label create "status:in-progress" --color "FBCA04" 2>/dev/null || true
gh label create "status:done" --color "0E8A16" 2>/dev/null || true
```

После создания каждого issue **запиши номер обратно в `KANBAN.md`** в колонку `Issue` (например `#42`).

Если `gh` недоступен — пропусти этот шаг и сообщи пользователю: «GitHub Issues не созданы (gh не настроен). Запусти `gh auth login` и `/kanban-sync` чтобы зеркалить вручную».

### Шаг 8. Краткий отчёт пользователю

Выведи:
- номер итерации, число задач, число модулей
- ссылки на созданные issues (если были)
- напоминание: задачи разбираются по веткам `T-XXX-slug`, после мержа PR с `Closes #N` запускай `/kanban-sync`

## Запреты

- НЕ создавай задачи, трогающие файлы из активных задач другой итерации
- НЕ принимай задачи без `files_touched` или без `acceptance`
- НЕ создавай отдельные файлы `iterations/I-XX/tasks/T-XXX.md` — всё живёт в `KANBAN.md`
- НЕ изменяй закрытые итерации в KANBAN.md
- НЕ коммить и не пуш без явной просьбы пользователя
